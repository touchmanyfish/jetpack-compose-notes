## 可组合函数

## 可组合项/可组合函数
理解为：
* 可组合项表示ui树上的一个节点，通过执行对应的可组合函数来生成和更新 
* “执行可组合项”：这个操作=执行对应可组合函数+一些其他的操作

## 重组
* [Recomposition](https://developer.android.com/develop/ui/compose/mental-model#recomposition)
```
 In Compose, you call the composable function again with new data. Doing so causes the function to be recomposed--the widgets emitted by the function are redrawn

// ...

Every time the button is clicked, the caller updates the value of clicks. Compose calls the lambda with the Text function again to show the new value; this process is called recomposition. Other functions that don't depend on the value are not recomposed.

As we discussed, recomposing the entire UI tree can be computationally expensive, which uses computing power and battery life. Compose solves this problem with this intelligent recomposition.

Recomposition is the process of calling your composable functions again when inputs change. When Compose recomposes based on new inputs, it calls only the functions or lambdas that might have changed, and skips the rest. By skipping functions or lambdas with unchanged parameters, Compose recomposes efficiently.
```

根据上述节选：
* "重组是处理调用可组合函数的过程":理解为，重组过程包含了调用可组合函数
* “函数被重组了”：理解为，这个函数对应的可组合项被执行，导致了它的重新执行
  

* [Lifecycle overview](https://developer.android.com/develop/ui/compose/lifecycle#lifecycle-overview)

```
正如管理状态文档中所述，一个组合将描述应用的界面，并通过运行可组合项来生成。组合是描述界面的可组合项的树结构。

当 Jetpack Compose 首次运行可组合项时，在初始组合期间，它将跟踪您为了描述组合中的界面而调用的可组合项。然后，当应用的状态发生变化时，Jetpack Compose 会安排重组。重组是指 Jetpack Compose 重新执行可能因状态更改而更改的可组合项，然后更新组合以反映所有更改。

组合只能通过初始组合生成且只能通过重组进行更新。重组是修改组合的唯一方式
```
理解为：
* “重组是指 Jetpack Compose 重新执行可能因状态更改而更改的可组合项，然后更新组合以反映所有更改” ：看来“重组”操作=执行可组合项(包含对应可组合函数执行)+更新组合操作

## 重启

### 重启作用域
* [状态读取](https://developer.android.com/develop/ui/compose/phases?hl=zh-cn#state-reads)
```
每个可以在读取状态发生更改时重新执行的代码块都是一个重启作用域。在不同阶段内，Compose 会跟踪状态 value 的更改和重启作用域。
```
编译器会代码包裹到一个匿名Function中，伪代码为（参考:[深入理解 Jetpack Compose SlotTable 系统](https://xie.infoq.cn/article/dd1e489d3152a4228a1257240)）:
```kotlin
 // Before compiler (sources)
@Composable
fun RestartGroupTest(str: String) {
    Text(str)
}

// After compiler
@Composable
fun RestartGroupTest(str: String, $composer: Composer<*>, $changed: Int) {
    $composer.startRestartGroup(-846332013)
    // ...
    Text(str)
    $composer.endRestartGroup()?.updateScope { next ->
        // 1. 
        RestartGroupTest(str, next, $changed or 0b1)
    }
}
```
前面所说的“重启作用域”大致对应（1.）处伪代码：  
1. 它捕获了传递给RestartGroupTest的参数
2. 执行这个lambda可以用相同的参数再次执行RestartGroupTest

### 理解

所以：
* 重启作用域的实质是一个包含了传递参数+方法调用的lambda
* “重启”指的通过调用可组合函数内部编译器生成的lambda，再次使用相同参数调用这个可组合函数 

## 不确定的地方

@Composable
fun RestartingDemo2() {
    Button(onClick = { name = System.currentTimeMillis().toString() }) {
        Text("button")

@Composable
fun ResChild2(value: Long) {
    println("value from parent:$value")
